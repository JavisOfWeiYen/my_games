1. Executive Summary

2026 年 Formula 1 是近年最大幅度的規則重置之一：底盤與動力單元規則同步改版，車體更小、更輕，導入全時主動空力，並改為更高比例的電能輸出。對賽車遊戲來說，2026 F1 的核心不是單純換車模，而是整個模擬架構都要支援「主動空力 + ERS 能量管理 + 新輪胎 + 新車隊 / 新 PU 格局」。

本文件建議採用「分層物理核心 + 規則版本化 + 內容資料驅動 + 混合 AI」架構。物理層處理輪胎、空力、ERS、燃油、碰撞與車輛動態；規則層處理排位、Sprint、輪胎義務、SC/VSC、旗號與判罰；AI 層則結合最佳路線、Utility AI、Behavior Tree 與局部 Reinforcement Learning。

最重要的工程原則

不要把規則與車輛數據寫死。所有規則、車隊、賽道、車輛、輪胎、ERS 都應資料化、版本化。

最重要的商業風險

真實 F1、車隊、車手、Logo、賽道外觀、官方圖像與計時資料都可能涉及授權。

最務實的開發策略

先做「規則真實、品牌虛構」版本；若取得授權，再替換為真實 F1 內容。

4. 2026 車輛技術架構與可用數據

2026 F1 的官方公開資料可以提供規則邊界，但各隊真實空力圖、扭力曲線、懸吊、重心、慣量、輪胎熱模型與可靠度多為非公開資料。因此遊戲資料應分成三種：official、estimated、calibrated。

資料項目公開程度遊戲建議
最小重量、尺寸、輪胎大方向高放入 regulation_version。
PU 架構、ERS-K 最大功率、燃油能量流限制中高作為硬限制，車隊可有不同效率參數。
空力 Cl / Cd / 主動模式低使用估算範圍 + calibration table。
扭力曲線與 ERS 部署地圖低用查表曲線 + 車隊係數。
懸吊、重心、慣量低以車隊 archetype 建立平衡差異。
輪胎熱衰與磨耗低建立 compounds 模型，由圈速與長距離資料校正。

6. 物理引擎設計

建議先用遊戲級 7DOF 車身模型：縱向、側向、偏航、heave、pitch、roll，加四輪輪速。輪胎使用 Pacejka 或 Brush 模型；空力使用 ClA/CdA 查表；ERS 與燃油做能量守恆與模式切換。

m * (du/dt - r*v) = ΣFx_tire - F_drag
m * (dv/dt + r*u) = ΣFy_tire
Iz * dr/dt        = ΣMz_tire + Mz_aero

F_drag      = 0.5 * rho * Cd(mode, yaw, steer) * A * V^2
F_downforce = 0.5 * rho * Cl(mode, rideHeight, pitch, yaw) * A * V^2

Fz_front = Fz_front_static + ΔFz_long + ΔFz_aero_front + ΔFz_pitch
Fz_rear  = Fz_rear_static  - ΔFz_long + ΔFz_aero_rear  - ΔFz_pitch

Fx = TyreLongitudinal(slip_ratio, slip_angle, Fz, temp, pressure)
Fy = TyreLateral(slip_angle, slip_ratio, Fz, camber, temp, pressure)

SoC(t+dt) = clamp(SoC(t) + η_harvest * E_harvest - E_deploy / η_deploy, SoC_min, SoC_max)
子系統最低可行高擬真版本
車身3DOF + 載重轉移7DOF + 懸吊幾何
輪胎Pacejka + 單一溫度胎面 / 胎體 / 輪圈多熱容模型
空力速度查表ride height / pitch / yaw 3D map
ERS每圈能量預算每個煞車區 / 直線段部署與回收地圖
碰撞區塊化損傷零件節點 + 二次故障 + debris

7. AI 車手與策略系統

建議不要只用一種 AI。高層使用 Behavior Tree 確保可解釋與守規；中層用 Utility AI 做風險/收益評估；局部超車、防守、輪對輪攻防可使用 RL 或 imitation learning。

AI 方法用途優點缺點
Behavior Tree守規、進站、旗號、SC/VSC可解釋、好除錯高互動戰術較僵硬
Utility AI超車收益、輪胎保護、ERS 節奏易調參、可平衡需人工設計特徵
RL局部攻防、封位、輪對輪互動性強訓練成本高、可能 exploit
Offline Optimal Racing Line基準路線、AI teacher穩定、可重現不處理即時對手互動

9. API 規格

MethodEndpoint用途
GET/api/regulations/{version}取得重量、尺寸、輪胎、ERS、比賽格式與判罰規則。
GET/api/seasons/2026/calendar取得賽曆、Sprint 標記、測試日程。
GET/api/teams/2026取得 2026 車隊、PU、底盤、授權狀態。
GET/api/cars/{car_id}取得單車模擬資料。
POST/api/sim/run-lap批次單圈模擬，用於 AI teacher 與參數校正。
POST/api/strategy/evaluate輸入輪胎、天氣、SC 機率，輸出進站策略。
POST/api/replay/encode產生 replay artifact。

10. Unity / Unreal 實作建議

Unity

適合快速原型、工具鏈、資料驅動與跨平台。建議物理核心用自訂 fixed tick，不完全依賴 WheelCollider。

FixedUpdate()
{
  input = ReadInput();
  vehicle.StepPhysics(input, fixedDeltaTime);
  aero.UpdateMode(vehicle.State);
  ers.Step(input, trackSegment);
  tyreModel.Step(vehicle.Wheels);
}

Unreal Engine

適合高畫質、車輛攝影機、多人連線與大型內容。建議用 Chaos 作為碰撞基礎，但高階賽車動態用自訂 movement component。

UF1VehicleMovementComponent::TickComponent()
{
  SampleInputs();
  StepTyres();
  StepAero();
  StepPowerUnitAndERS();
  IntegrateRigidBody();
  PublishTelemetry();
}

13. 授權與風險

若要使用真實 Formula 1、車隊、車手、Logo、官方圖像、車身塗裝、賽道外觀、官方計時資料，通常需要授權。最務實的產品策略是先做「規則真實、品牌虛構」版本，保留資料結構中 licensed_content 欄位，未來取得授權後再替換為真實內容。

內容風險替代方案
F1 / Formula 1 / Grand Prix 名稱與 Logo高使用虛構聯盟名稱。
真實車隊與車手高使用虛構隊名、虛構車手與自製塗裝。
真實賽道輪廓與場館中高創作受啟發但非 1:1 的賽道。
官方 TV graphics / HUD高自製 HUD 視覺語言。
官方計時與賽果資料中高使用自產模擬資料或取得資料授權。
